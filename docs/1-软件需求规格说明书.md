## 一、引言

### 1.1 项目背景
在日常信息分享中，长链接（如会议链接、文档链接）在群聊、短信等场景下占用大量字符，影响阅读体验，以及微博、短信等平台都有字数限制，只能缩减信息内容。现有短链接服务虽多，但自建一个轻量、高性能的短链接服务能更好地理解核心原理，并锻炼后端开发能力。

### 1.2 项目目标
- 实现短链接生成、重定向、点击计数等核心功能。
- 集成 Redis 缓存提升热点链接访问速度。
- 增加限流保护与异步统计写入，应对高并发场景。
- 提供 Docker 一键部署，保证可移植性。

### 1.3 适用范围
本文档适用于短链接服务的需求分析、系统设计、编码、测试及部署阶段，作为项目开发依据。

### 1.4 用户角色
- 普通用户：通过 Web 界面或 API 生成短链接、访问短链接。
- 开发者：维护服务、配置参数、查看日志和监控。

### 1.5 假定与约束
- **假定 1**：后端依赖 MySQL 和 Redis，用户提供有效的长 URL（协议头为 http:// 或 https://）。
- **约束 1**：本版本暂不实现用户登录、多租户、分布式部署等特性。
- **约束 2**：开发语言 Go 1.25+，Web 框架 Gin，存储使用 MySQL 8.0+，缓存使用 Redis 7.2+，部署使用 Docker。
- **约束 3**：生成短链时，若用户输入无效 URL（如不可访问或格式错误），后端应进行有限次重试（例如最多 3 次，间隔 1 秒），仍失败则返回 400 错误并提示“无效链接”。
- **约束 4**：短链有效期（FR-07）为可选功能，若时间不足可暂缓实现，不影响核心功能交付。

## 二、功能性需求

| 编号  | 需求描述                                                                                                                             | 优先级 |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------ | ------ |
| FR-01 | **短链接生成**：接收长 URL，生成唯一短码，返回短链接（格式如 http://short.xxx/abc123 ）。                                               | 高     |
| FR-02 | **重定向**：访问短链接时，根据短码查询原始长 URL，返回 HTTP 302 临时重定向（确保每次访问都能计数）。                                                           | 高     |
| FR-03 | **点击计数**：每次重定向成功，更新该短链的点击次数。                                                                 | 高     |
| FR-04 | **Redis 缓存**：热点短链（高频访问）缓存到 Redis，降低数据库压力。缓存过期时间可配置（如 1 小时）。 | 高     |
| FR-05 | 	**限流保护**：基于 IP 或用户 ID，限制短链生成频率（例如每分钟最多 5 次），使用令牌桶或漏桶算法。                      | 中     |
| FR-06 | **异步统计写库**：点击计数不直接写数据库，而是先写入 Redis，后台定时批量回写 MySQL，提升高并发性能。               | 中     |
| FR-07 |**短链有效期**（可选）：支持设置短链有效期（如 7 天），过期后自动失效或进入回收站。                           | 低     |

## 三、非功能性需求

| 编号   | 需求描述                                                                                                                | 指标 / 约束                              |
| ------ | ----------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| NFR-01 | **性能**：单机 QPS ≥ 5000（重定向请求），平均响应时间 ≤ 20ms。                         | 使用 go-wrk 压测验证                 |
| NFR-02 | **可靠性**：Redis 或 MySQL 短暂不可用时，服务应降级（如直接查数据库或返回错误），不崩溃。同时，生成短链时对用户输入的 URL 进行有限重试（例如最多 3 次），避免因临时网络问题导致失败。                                  | 熔断或重试机制 |
| NFR-03 | 	**可观测性**：结构化日志（JSON 格式），记录每个请求的短码、耗时、客户端 IP。                             | 使用 slog                         |
| NFR-04 | **安全性**：<br> - 限流防止恶意生成<br> - 使用参数化查询（GORM）防止 SQL 注入<br> - 防止缓存穿透：对不存在短码的请求，直接返回 404 Not Found，不缓存空值。 | 无额外库依赖 |
| NFR-05 | **可移植性**：提供 Dockerfile 和 docker-compose，一键启动 MySQL、Redis、短链接服务。 | 镜像大小 ≤ 50MB        |
