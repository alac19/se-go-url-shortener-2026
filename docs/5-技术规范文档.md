## 三、代码规范

### 3.1 命名规范

| 类型       | 规则                           | 示例                          |
| ---------- | ------------------------------ | ----------------------------- |
| 包名       | 小写、单数、无下划线           | `handler`, `service`          |
| 结构体     | 大驼峰                         | `ShortLinkService`            |
| 接口       | 大驼峰，单方法可加 `-er`       | `LinkRepository`              |
| 函数/方法  | 大驼峰（导出）或小驼峰（私有） | `GenerateShortLink`, `newRepo`|
| 变量       | 大驼峰（导出）或小驼峰（私有）                         | `shortCode`, `LongURL`        |
| 常量       | 大驼峰或全大写                 | `MaxRetries`                  |
| 文件名     | 小写，下划线分隔               | `shortlink_test.go`           |
| 缩写词     | 对于导出的类型、字段、函数名中的缩写词（URL、HTTP、API、IP、ID 等），保持全大写；对于未导出的私有变量或局部变量，使用全小写 | `LongURL`, `HTTPClient`, `UserID`, `url`, `httpClient`, `id`           |

### 3.2 注释规范

- 每个导出的符号必须有文档注释，以符号名开头。
- 包注释放在包声明上方，说明包的作用。

### 3.3 错误处理

- 使用 `fmt.Errorf` 或 `errors.New` 创建错误。
- 在 service/repository 层向上返回明确错误。
- 在 handler 层根据错误类型设置 HTTP 状态码。

### 3.4 代码风格

- 使用 `go fmt` 自动格式化。
- 每行长度不超过 120 字符（建议）。
- 禁止使用 `panic`（除初始化失败外）。
- 导出函数按逻辑分组，使用空行分隔。

## 四、配置规范

### 4.1 配置文件格式

使用 TOML，示例 `configs/config.toml`：

```toml
[mysql]
dsn = "root:123456@tcp(mysql:3306)/shortlink_db?charset=utf8mb4&parseTime=True&loc=Local"

[redis]
addr = "redis:6379"
password = ""
db = 0

[server]
port = 8080
domain = "http://localhost:8080/"

[ratelimit]
every_seconds = 12
burst = 5

[asyncflush]
interval_seconds = 30
scan_count = 100

[urlcheck]
timeout_seconds = 3
max_retries = 3
retry_delay_seconds = 1

[cache]
ttl_seconds = 3600

[log]
level = "info"
file_path = "logs/shortlink.log"
```

### 4.2 配置加载

- 使用 `toml.DecodeFile` 加载。
- 加载后执行 `Validate()` 校验参数合法性。

## 六、测试规范

### 6.1 单元测试

- 使用表驱动测试。
- 使用 mock 结构体隔离外部依赖。
- 覆盖率目标：核心模块 ≥ 85%。

### 6.2 集成测试

- 手工执行 `curl` 验证完整流程。
- 测试用例覆盖生成、重定向、限流、格式校验、异步统计等。

### 6.4 压测

- 使用 `go-wrk` 或 `ab` 对重定向接口进行压测，记录 QPS 和延迟。

## 七、构建与部署规范

### 7.1 Docker 镜像
多阶段构建，最终镜像约 20MB。

### 7.2 Docker Compose
- 一键启动 MySQL、Redis、短链接服务。
- 使用标准命名 `docker-compose.yml`，确保用户可直接执行 `docker-compose up` 而无需额外参数。

### 7.3 版本管理
使用 `git tag` 标记版本，如 `v1.0.0`。

## 八、版本控制规范

### 8.1 分支策略

- **main**：稳定版，只接受 `develop` 的 PR。
- **develop**：开发集成分支。
- **feature/xxx**：新功能分支。
- **hotfix/xxx**：紧急修复分支。

### 8.2 提交信息规范

```
<type>: <subject>

[optional body]
```

- `type`：feat, fix, docs, style, refactor, test, chore。
- `subject`：简短描述，不超过 50 字符。

示例：
```
feat: 实现短链生成接口
fix: 修复 Redis 连接池泄漏
```
