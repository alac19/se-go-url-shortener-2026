## 一、测试概述

### 1.1 测试目标
验证短链接服务在功能、性能、可靠性方面满足需求规格说明书的要求，确保以下核心能力：
- 短链接生成（POST /api/links）
- 重定向（GET /{code}）及点击计数
- Redis 缓存与缓存旁路模式
- 限流保护（令牌桶，基于 IP）
- 异步统计写入（Redis → MySQL 批量迁移）
- URL 格式校验与可达性重试
- 结构化日志（slog）与配置管理
- 高并发下重定向接口的吞吐量和延迟表现

### 1.2 测试范围

| 测试类型 | 覆盖范围 |
| -------- | -------- |
| 单元测试 | base62 编码、urlcheck 包、limiter 包、logger包、service 层（核心业务逻辑） |
| 集成测试 | 端到端功能测试（手工 curl / Invoke-RestMethod） |
| 性能测试 | 不同并发下的 QPS、平均延迟、P99 延迟、错误率；长时间稳定性测试 |
| 安全性测试 | 限流、输入校验、SQL 注入防护（GORM 自动参数化） |

## 三、测试策略

### 3.1 测试方法
- **单元测试**：使用 Go 标准 `testing` 包，运行 `go test -v -cover ./...` 获取覆盖率和结果。
- **集成测试**：通过手工执行 `curl` 或 `Invoke-RestMethod` 验证完整业务流程（生成短链、重定向、限流、格式校验等）。
- **性能测试**：使用 `go-wrk` 对重定向接口进行压测，记录 QPS、平均延迟、P99 延迟、错误率；每个并发级别执行 3 次取平均值；每次测试后等待 1 分钟避免热累积。同时进行长时间稳定性测试（如持续 5 分钟）。
- **安全性测试**：验证限流返回 429、格式错误返回 400、SQL 注入尝试（GORM 自动防护）。

### 3.2 测试通过标准
- **单元测试**：所有测试用例通过，核心模块（base62, urlcheck, limiter, service）代码覆盖率 ≥ 80%。
- **集成测试**：所有功能需求（生成、重定向、统计、缓存、限流、校验）验证通过，无功能性缺陷。
- **性能测试**：重定向接口 QPS ≥ 5000，平均延迟 ≤ 20ms，P99 延迟 ≤ 50ms，错误率 = 0%。
- **安全性测试**：限流正确拦截超频请求；无效 URL 返回 400；无法注入 SQL。

## 四、测试用例

### 4.1 单元测试用例

| 模块    | 测试内容                                                                 | 验证点                         | 覆盖率目标 |
| ------- | ------------------------------------------------------------------------ | ------------------------------ | ---------- |
| base62  | 正常数字转换、边界值（0, 1, 61, 62）                           | 编码唯一且符合字符集           | 100%       |
| urlcheck| URL 格式校验（合法/非法）；可达性检查（模拟服务器：200/404/405/超时等）  | 返回正确错误类型，重试逻辑正确 | ≥ 90%      |
| limiter | 单 IP 限流、多 IP 独立、令牌恢复、并发安全（-race）                      | Allow 返回值正确               | 100%       |
| service | CreateShortLink（正常/数据库错误）、Redirect（缓存命中/未命中/降级）、FlushStats（批量写入成功/失败） | 业务逻辑正确，错误处理得当     | ≥ 85%      |
| logger  | 日志级别解析、文件创建、默认输出                                         | 初始化无 panic，文件可写       | ≥ 80%      |

### 4.2 集成测试用例（手工执行）

> 以下用例通过 `curl` 或 `Invoke-RestMethod` 执行，验证系统整体功能。在服务启动后依次测试，并记录结果。

| 编号 | 测试项             | 操作步骤（示例命令）                                                                                                                                                     | 预期结果                                                      |
| ---- | ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------- |
| IT01 | 生成短链（正常）   | `curl -X POST http://localhost:8080/api/links -H "Content-Type: application/json" -d '{"url":"https://example.com"}'`                                                   | 返回 JSON 包含 `short_url`，状态码 200                        |
| IT02 | 生成短链（格式错误）| `curl -X POST http://localhost:8080/api/links -H "Content-Type: application/json" -d '{"url":"example.com"}'`                                                            | 返回 `{"error":"invalid url format"}`，状态码 400             |
| IT03 | 生成短链（不可达） | `curl -X POST http://localhost:8080/api/links -H "Content-Type: application/json" -d '{"url":"http://.com"}'`                                                            | 返回 `{"error":"url not reachable after retry"}`，状态码 400  |
| IT04 | 重定向成功         | 先通过 IT01 获得短码（如 `abc123`），然后 `curl -v http://localhost:8080/abc123`                                                                                         | 响应头 `Location` 指向长链接，状态码 302                      |
| IT05 | 重定向（短码不存在）| `curl -v http://localhost:8080/notexist`                                                                                                                                | 返回 404 及 `{"error":"短链不存在"}`                          |
| IT06 | 限流验证           | 连续快速发送 6 次 POST 请求（限流配置：每12秒5个令牌），观察响应                                                                                                         | 前5次成功，第6次返回 429 `{"error":"too many requests"}`     |
| IT07 | 缓存验证           | 首次访问短链后，检查 Redis 中是否存在 `shortlink:短码` 键（例如 `docker exec -it redis-dev redis-cli GET shortlink:abc123`）                                            | 键存在且值为长链接                                            |
| IT08 | 统计计数（异步）   | 多次访问同一短链，等待 30 秒（默认异步间隔 5 秒，增加以便观察），观察 MySQL 中 `click_count` 字段是否累加，Redis 中 `stats:短码` 键是否被删除                                                | 数据库计数增加，Redis 统计键消失                              |

### 4.3 性能测试用例

> 使用 `go-wrk` 工具，目标短链需提前生成（例如 `http://localhost:8080/abc`）。所有测试在 Linux 虚拟机或 WSL2 中运行，以避免 Windows 网络栈性能偏差。

| 编号 | 并发数 `-c` | 总请求数 `-n` | 持续时间 `-d` | 测试命令                                                                 | 记录指标               |
| ---- | ----------- | ------------- | ------------- | ------------------------------------------------------------------------ | ---------------------- |
| PT01 | 100         | 50000         | -             | `go-wrk -c 100 -n 50000 http://localhost:8080/abc`                       | QPS、平均延迟、P99、错误率 |
| PT02 | 200         | 100000        | -             | `go-wrk -c 200 -n 100000 http://localhost:8080/abc`                      | 同上                   |
| PT03 | 500         | 250000        | -             | `go-wrk -c 500 -n 250000 http://localhost:8080/abc`                      | 同上                   |
| PT04 | 100         | -             | 300s          | `go-wrk -c 100 -d 300 http://localhost:8080/abc`                         | 稳定性：QPS、延迟波动、错误率 |
| PT05 | 200         | -             | 300s          | `go-wrk -c 200 -d 300 http://localhost:8080/abc`                         | 同上                   |

**说明**：
- 每个并发级别可执行 3 次取平均值。
- 服务端需运行在 `release` 模式（`gin.SetMode(gin.ReleaseMode)`），并关闭调试日志。
- 压测前确保 Redis 和 MySQL 已启动，且短链已缓存（首次访问后）。

### 4.4 安全性测试用例

| 编号 | 测试项               | 操作步骤                                                                 | 预期结果                                         |
| ---- | -------------------- | ------------------------------------------------------------------------ | ------------------------------------------------ |
| ST01 | 限流防刷             | 同 IT06，快速发送超过阈值的请求                                           | 返回 429                                         |
| ST02 | SQL 注入             | 生成短链时 `url` 字段尝试 `' OR 1=1 --`                                   | 不会导致 SQL 错误，应返回格式错误或不可达        |
| ST03 | 恶意长链（JavaScript）| `javascript:alert(1)`                                                    | 格式校验失败，返回 400                           |
| ST04 | 超长 URL             | 发送长度 > 2000 字符的 URL                                                | 数据库 TEXT 字段可存储，应正常处理或返回 400（视业务） |

## 五、测试执行计划

1. **单元测试执行**：运行 `go test -v -cover ./...`，记录每个包的通过情况和覆盖率，截图保存。
2. **集成测试执行**：按照 IT01~IT08 顺序执行，使用 `curl` 命令，保存输出日志或截图。
3. **性能测试执行**：
   - 启动服务（确保 `release` 模式）。
   - 生成一个测试短链（例如 `abc123`）。
   - 在 Linux 虚拟机/WSL2 中依次执行 PT01~PT03（短时压测），每个级别运行 3 次，记录输出。
   - 执行 PT04~PT05（稳定性测试），运行 5 分钟，观察 QPS 和延迟变化，记录服务端资源占用。
4. **安全性测试执行**：按照 ST01~ST04 顺序执行，验证防护是否生效。
5. **测试报告编写**：汇总所有结果，形成最终报告（包含环境、数据、结论）。
